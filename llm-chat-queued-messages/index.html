Queued Messages in LLM Chat Apps

Design research · 2026-05-18

TL;DR

None of the big LLM chat apps (ChatGPT, Claude, Gemini, Copilot) ship native message queuing — the consistent native pattern is disable send + show stop button while streaming, and users have built a Chrome-extension cottage industry to fill the gap. The emerging "right answer" pattern, validated by ChatGPT Queue (extension) and Cursor (feature request hotly upvoted): let the user keep typing, hold pending messages as ghost-styled bubbles right above the composer, allow edit/delete before they fire, then drain the queue sequentially when streaming completes.

Recommendations / Next Steps

  1. Always-typeable composer. Never disable the textarea while streaming. Pressing Enter while a response is in flight should enqueue, not no-op. The send icon morphs into a "queue" icon (e.g. ⏱ or +) to signal the change in behavior. ChatGPT, Claude, and Gemini all currently block this — that's the gap to exploit.
  2. Queued messages live above the composer as ghost bubbles. Don't insert them into the chat thread (that lies about send order and conflates "sent" with "queued"). Render them as a stack of muted/dashed-border user bubbles directly above the input, with each row showing the text, an edit pencil, and a delete ×.
  3. Stop ≠ Cancel queue. The streaming stop button cancels the current generation only. Queued items should still fire. Provide a separate "clear queue" affordance so users can abort the pipeline without aborting the current response.
  4. Drain visibly, not invisibly. When streaming ends, animate the top queued bubble sliding down into the chat as the new user turn. Silent insertion makes queued messages feel lost.
  5. Per-message status icons (small). Tiny glyph inside each queued bubble: ⏱ queued, ↑ sending, ⏵ streaming response. The bubble's position (above composer vs. in thread) is already 90% of the signal.
  6. Cap the queue and warn at the limit. 5 is plenty. Unbounded queues invite stream-of-consciousness Enter-mashing and a wall of robotic answers the user didn't actually want.
  7. Make queued context visible in the prompt sent to the model. When message #2 fires, prepend a note like "User also queued these follow-ups while you were responding: …" — lets the model batch or reorder.

Key Examples

Patterns

Universal in 2026 native LLM chats (ChatGPT, Claude, Gemini, Copilot, Perplexity):

  • Send button morphs into stop (■) the moment streaming begins.
  • Textarea remains editable but Enter is a no-op until streaming ends. (This is the friction users are complaining about.)
  • The assistant's in-progress bubble shows a caret/cursor blink or shimmer to signal liveness.
  • No queue, no buffer — if you press Enter early it's lost.

Common across queue-enabling extensions and agent IDEs (ChatGPT Queue, Cursor, agentic coding tools):

  • Queue appears above the composer, not in the thread.
  • Per-item edit + delete affordances.
  • Sequential drain (FIFO).
  • Visual differentiation from sent messages: dashed border, lower opacity, distinct background.

Anti-Patterns

  • Disabling the textarea entirely. Kills flow-of-thought capture.
  • Inserting queued messages into the chat thread immediately. Confuses send order and breaks the model's view of conversation.
  • Auto-merging queued messages into one big prompt. Tempting, but changes user intent. Send sequentially.
  • Silent queue drain. Users wonder if they accidentally sent it again. Animate the transition.
  • Hiding queue length. Always show count.
  • Conflating "stop generating" with "clear queue". Different intents, different buttons.

Unique Angles

  • Cursor's "background agent" framing. Treats long-running tasks as something you walk away from — so queued messages double as a backlog you populate while the agent works. Reframing queue as intentional planning (vs. waiting room) opens different patterns: numbered todos, drag-to-reorder, even a "send all as one plan" merge button as an explicit choice.
  • The status-log composer. Bitrig blends progress events ("renamed project", "edited code") into the area right above the composer. Exactly the context you want visible while queueing — you can reference newly-renamed files in your follow-up.
  • Cap the queue, not just for UX but for cost. Each queued message is a full turn — easy way for a user to run up a $5 Opus bill by mashing Enter.

Findings

The research landscape is asymmetric: near-zero native support, heavy unmet demand.

  • OpenAI forum thread "Feature Request: Message queue while GPT is generating responses" remains open and active in 2026.
  • Cursor's "Queue Agent Messages" feature request is a recurring forum thread; Cursor has partially shipped it as "Add follow-up".
  • Multiple Chrome extensions (ChatGPT Queue, ChatGPT Message Queue, queue-chatgpt) exist specifically to retrofit this onto ChatGPT.
  • The Lazyweb screenshot corpus has no clean examples of queued state in any major LLM chat. That absence is itself the finding: this is greenfield UX.

Adjacent precedent worth borrowing:

  • iMessage's send-while-offline behavior: outbox messages render in the thread but greyed out with a small clock icon. Lift the visual treatment (muted color, status glyph) without lifting the placement.
  • Slack's draft system: visibility model — small indicator near input, not in thread — is what we want.
  • Email outbox: same primitive applied to a different bottleneck (model latency instead of network).
